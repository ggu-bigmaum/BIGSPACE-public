## 🔴 CRITICAL

### C1 — Shell Injection (routes.ts:761,784)

**한 줄 요약:** 업로드된 파일명이 서버 터미널 명령어로 둔갑할 수 있음

**설명:**
ZIP 파일 업로드 시 서버가 터미널 명령어(`unzip`)를 직접 실행함.
파일명에 악성 코드를 심으면 서버 전체 파일 삭제 같은 명령이 함께 실행될 수 있음.

**현실적 위험도:** 로그인한 사용자만 업로드 가능 → 완전 개방은 아님.
단, 계정 탈취 or 내부자 위협 시 서버 전체가 위험.

**해결 방향:** `execSync("unzip ...")` → `adm-zip` 라이브러리 교체 (터미널 명령어 제거)

---

### C2 — 첫 유저 Admin 승격 레이스컨디션 (routes.ts:88-92)

**한 줄 요약:** 동시에 2명이 가입하면 둘 다 관리자가 될 수 있음

**설명:**
"첫 번째 가입자 = 자동 admin" 로직이 있는데,
동시에 2개 요청이 들어오면 둘 다 "나 첫 번째야"라고 인식해서 둘 다 admin 됨.

**현실적 위험도:** 서비스 초기 1회만 발생 가능한 상황. 이미 admin 계정이 있으면 발생 안 함.

**해결 방향:** DB 트랜잭션 또는 유니크 제약으로 "admin이 0명일 때만" 원자적으로 처리

---

### C3 — OAuth 센티넬 비밀번호 (routes.ts:165)

**한 줄 요약:** Google 로그인 계정으로 일반 비밀번호 로그인이 가능한지 검증 필요

**설명:**
Google OAuth로 가입한 계정은 비밀번호 없이 만들어짐.
대신 내부적으로 `"!oauth-no-local-login!"` 이라는 더미 문자열을 비밀번호 칸에 넣어둠.
만약 이 문자열이 실제 로그인 검증을 통과한다면, 누구나 Google 계정을 일반 로그인으로 접근 가능.

**현실적 위험도:** ✅ 실제로 안전함 — 코드 확인 결과 `randomUUID()`로 매번 다른 랜덤 해시를 비밀번호로 저장. 더미 문자열 방식 아님.

**해결 방향:** 불필요. 이미 안전하게 구현되어 있음.

---

### C4 — FK 없음 (schema.ts)

**한 줄 요약:** 레이어 삭제해도 연결된 데이터가 DB에 그대로 남음

**설명:**
레이어를 지우면 그 레이어에 딸린 피처(점/선/면 데이터), 캐시들이 같이 지워져야 하는데
DB 레벨에서 "같이 지워라" 규칙(FK CASCADE)이 없었음.
→ 레이어는 사라지고 데이터는 떠도는 고아 레코드 발생.

**현실적 위험도:** DB 용량 낭비, 쿼리 오염. 당장 크래시는 없지만 장기적으로 문제.

**해결 방향:** ✅ 이미 해결됨 — commit `3add5dd`에서 전체 FK + CASCADE 적용 완료.

---

### C5 — Geometry가 PostGIS 타입 아닌 JSONB (schema.ts)

**한 줄 요약:** 지도 데이터를 비효율적인 방식으로 저장 중 → 공간 쿼리 느림

**설명:**
지도의 점/선/면 데이터(Geometry)를 PostGIS 전용 타입이 아닌 JSONB(일반 텍스트 덩어리)로 저장 중.
PostGIS 타입으로 저장하면 "이 지역 안에 있는 점 찾기" 같은 공간 쿼리에 R-tree 인덱스(초고속 검색)를 쓸 수 있음.
지금은 전체 데이터를 다 뒤지는 풀스캔 방식.

**현실적 위험도:** ✅ 실제로 안전함 — Opus가 실제 DB 확인 결과, `geom` 컬럼(PostGIS 타입) + GiST 인덱스가 이미 존재. 실제 쿼리도 `ST_Intersects`, `ST_AsMVTGeom` 등 PostGIS 네이티브 사용 중.

**진짜 문제:** Drizzle 스키마에 `geom` 컬럼이 선언 안 됨 → insert 시 JSONB에 먼저 넣고 별도 UPDATE로 geom 동기화하는 2단계 구조. 기능은 동작하지만 구조가 지저분함.

**해결 방향:** 급하지 않음. Drizzle에 custom PostGIS 타입 등록으로 정리 가능.

## 🟠 HIGH

### H1 — WMS/WFS 프록시 오픈 릴레이 (routes.ts:496-552)

**한 줄 요약:** 로그인 없이 우리 VWorld API 키를 무한정 소비 가능

**설명:**
`/api/proxy/wms` 엔드포인트가 인증 없이 열려있음.
외부인이 이 주소를 알면 우리 서버를 통해 VWorld에 무제한 요청 가능.
키 자체는 안 보이지만 키 사용량(할당량)을 소진시킬 수 있음.

**현실적 위험도:** 인터넷 공개 서버라면 위험. 사내 전용이면 낮음.

**해결 방향:** `requireAuth` 미들웨어 추가 (한 줄). 로그인 필수 서비스이므로 바로 적용 가능. 난이도 최하.

---

### H2 — Rate Limiting 없음 (routes.ts 전체)

**한 줄 요약:** 로그인 시도 횟수 제한이 없어 비밀번호 무한 대입 가능

**설명:**
로그인/회원가입 API에 요청 횟수 제한이 없음.
봇으로 1초에 수천 번 비밀번호 시도 가능.

**해결 방향:** `express-rate-limit` 패키지로 IP당 요청 횟수 제한

---

### H3 — WFS 메모리 누수 (useLayerRenderer.ts:252-287)

**한 줄 요약:** WFS 레이어 데이터가 지도에 계속 쌓이고 안 지워짐

**설명:**
WFS 레이어를 껐다 켜면 이전 데이터를 지우지 않고 위에 새 데이터를 쌓음.
오래 쓸수록 브라우저 메모리 점유 증가, 결국 느려지거나 크래시.

**해결 방향:** 레이어 갱신 전 `source.clear()` 호출

---

### H4 — OL 레이어 unmount 시 미정리 (useLayerRenderer.ts)

**한 줄 요약:** 레이어 삭제해도 지도 객체가 메모리에 남음

**설명:**
React에서 레이어 컴포넌트가 사라질 때 OpenLayers 레이어 객체를 지도에서 제거하지 않음.
H3과 합쳐서 장시간 사용 시 메모리 누수 누적.

**해결 방향:** useEffect cleanup 함수에서 `map.removeLayer()` 호출

---

### H5 — fetchAndRenderLayer 레이스컨디션 (useLayerRenderer.ts:111)

**한 줄 요약:** 지도 이동 중 빠르게 움직이면 순간 빈 화면 노출

**설명:**
데이터를 지운 다음 새 데이터를 가져오는 사이에 짧은 공백이 생김.
빠른 이동 시 깜빡임 또는 빈 지도 노출.

**해결 방향:** 새 데이터 준비 완료 후 교체 (더블 버퍼링 패턴)

---

### H6 — deleteLayer/createFeatures 트랜잭션 없음 (storage.ts:113,153)

**한 줄 요약:** 레이어 삭제/생성 중 서버가 죽으면 절반만 처리된 상태로 남음

**설명:**
레이어 삭제나 피처 대량 삽입이 여러 단계로 나뉘어 실행됨.
중간에 오류나면 일부는 삭제, 일부는 남은 불일치 상태 발생.

**해결 방향:** DB 트랜잭션으로 묶어서 "전부 성공 or 전부 롤백" 처리

---

### H7 — 캐시 테이블 유니크 제약 없음 (schema.ts)

**한 줄 요약:** 동시에 캐시 빌드하면 중복 데이터 삽입됨

**설명:**
경계 집계 캐시, 격자 캐시 테이블에 유니크 제약이 없음.
동시에 2개 빌드 작업이 돌면 같은 데이터가 2번 들어감.

**해결 방향:** 유니크 인덱스 추가 + ON CONFLICT DO NOTHING

---

### H8 — 감사로그 엔드포인트 admin 가드 없음 (routes.ts:194)

**한 줄 요약:** 일반 유저도 전체 감사로그(IP, User-Agent 등) 조회 가능

**설명:**
감사로그는 관리자만 봐야 하는 민감 정보인데, 로그인만 하면 누구나 볼 수 있음.

**해결 방향:** `requireAdmin` 미들웨어 추가 (한 줄)
